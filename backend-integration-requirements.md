# Navin Frontend → Backend Integration Requirements

This document details everything the **navin-backend** must provide to support the current Navin frontend codebase. It was generated by analyzing the frontend repository (`Navin-xmr/navin-frontend`) and enumerating all API contracts, data shapes, authentication rules, file-upload requirements, real-time expectations, and third-party dependencies the frontend assumes.

> **Scope:** This is a contract-first analysis. It does not prescribe a specific backend language or framework, but it maps out the routes, methods, request/response payloads, authorization roles, and external services the backend must implement.

---

## 1. High-Level Architecture Assumptions

The frontend expects the backend to be a **REST API** with the following characteristics:

- **Base URL** configured via `VITE_API_BASE_URL` (e.g. `http://localhost:3000/api`).
- **JSON request/response bodies** with a `{ data: ... }` envelope for most endpoints (some endpoints return the raw paginated object — see per-route notes).
- **Cookie-based sessions** (`withCredentials: true`) plus a **JWT bearer token** stored in `localStorage` under `authToken`.
- **MongoDB-style `_id` fields** on resources; the frontend normalizes `_id` → `id` automatically.
- **File upload support** for images, PDFs, and generic documents via `multipart/form-data`.
- **CORS enabled** for the frontend origin with credentials support.
- **Real-time events** delivered via Server-Sent Events (`/api/events`) with a polling fallback (`/api/events/poll`).
- **Stellar Soroban smart-contract integration** for escrow/settlement state; the backend is expected to persist on-chain references (`stellarTokenId`, `stellarTxHash`) and mirror contract state.

## 2. Environment Variables the Backend Must Expose / Accept

The frontend reads these `VITE_*` variables at build/run time. The backend must provide compatible values and infrastructure:

| Variable | Purpose | Backend Implication |
|----------|---------|---------------------|
| `VITE_API_BASE_URL` | REST API root | Backend must listen on the configured host/port and mount all routes under `/api`. |
| `VITE_STELLAR_NETWORK` | `testnet` or `mainnet` | Backend must sign/submit Soroban transactions to the matching network. |
| `VITE_SOROBAN_RPC_URL` | Soroban RPC endpoint | Backend must use the same RPC for simulation/submission. |
| `VITE_ESCROW_CONTRACT_ID` | Deployed Soroban escrow contract | Backend must deploy, store, and call this contract. |
| `VITE_SENTRY_DSN` | Frontend error tracking | Optional for backend, but backend should return structured errors Sentry can ingest. |

Additional backend-only secrets/variables the backend will likely need:

- `JWT_SECRET` / `JWT_PRIVATE_KEY` — for signing `authToken`.
- `MONGODB_URI` — the frontend explicitly normalizes MongoDB `_id` to `id`.
- `REDIS_URL` — recommended for sessions, rate-limiting, and SSE fan-out.
- `AWS_S3_BUCKET` / `S3_ENDPOINT` / `CLOUDINARY_*` — for file/blob storage.
- `SMTP_HOST`, `SMTP_USER`, `SMTP_PASS` — for invitation, password-reset, and notification emails.
- `TWILIO_*` or equivalent — for SMS OTP verification.
- `STELLAR_SECRET_KEY` / `HSM` — for backend signing of Soroban transactions (if backend is the escrow operator).

## 3. Authentication & Authorization

### 3.1 Auth Flow

- Login/Signup/Company registration return `{ data: { user, token } }`.
- The frontend stores `token` in `localStorage` and sends it as `Authorization: ******
- The frontend decodes the JWT with `jose` to extract `role`, `userId` (`sub` or `userId`), and `exp`.
- `POST /auth/refresh` returns `{ data: { token } }`.
- `POST /auth/logout` clears the token on the frontend.

### 3.3 Role Model

The frontend operates with **two top-level account personas**:

- `company` — logistics provider / enterprise.
- `customer` — receiver / shipper.

Within a company, users have internal RBAC roles:

- `Admin`
- `Manager`
- `Viewer`
- `Driver`

The backend must:

- Issue JWTs containing the top-level role (`company` or `customer`).
- Enforce company-internal role checks for endpoints such as user management, API keys, analytics, settlements, and shipment creation.
- Support the permission matrix implied by `src/utils/rbac.ts`:
  - `company` can: create shipments, upload proof, confirm milestones, release payments, dispute settlements, manage team, view analytics, manage API keys.
  - `customer` cannot perform any of those actions.

## 16. Stellar / Soroban Blockchain Integration

### 16.1 Frontend Blockchain Services

- `services/soroban/client.ts` — submits transactions to Soroban RPC.
- `services/soroban/escrow.ts` — calls the escrow smart contract.
- `services/stellar/adapters/` — wallet adapters for Freighter, Albedo, and LOBSTR.

### 16.2 Backend Responsibilities
